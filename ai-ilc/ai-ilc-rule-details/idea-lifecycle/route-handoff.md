# Stage 6: Route & Handoff

**Lead persona:** `#persona-product-manager`
**Sub-role:** `#persona-subrole-change-manager` (impact assessment, organizational readiness, adoption scope)
**Execution:** CONDITIONAL — only if Stage 5 (Approve) = APPROVED
**Purpose:** Determine where this approved idea goes and produce the right brief for that destination — with zero context loss at handoff.

---

## Why This Stage Exists

An approved idea without routing is a decision without action. This stage answers: "Now that we've said YES — what happens next?" The answer depends on whether a project exists and whether the idea's impact is large or small. The brief produced here carries forward everything learned during shaping, evaluation, and scoping — so the successor never starts cold.

---

## Depth Adaptation

| Depth | Routing Behavior |
|-------|-----------------|
| **Minimal** | Quick routing determination (2 questions) + concise brief (1-2 pages). |
| **Standard** | Full routing with impact assessment + structured brief (2-4 pages). |
| **Comprehensive** | Full routing + detailed impact analysis + comprehensive brief with stakeholder map and transition considerations (4-6 pages). |

---

## Step-by-Step Execution

### Step 1: Load Context

1. Read `ilc-state.md` — confirm idea is Approved, load all accumulated context
2. Load: Idea Statement, Score, Value Analysis, Scope Summary, Decision Record
3. Load any approval conditions from Stage 5
4. Activate the Change Manager sub-role: think about impact, disruption, stakeholders, adoption

### Step 2: Determine Project Existence

```markdown
### Q-{nn}: Does a project already exist for this?

**Context:** This determines where your approved idea goes.
- If NO project exists → this becomes a new project (feeds AI-PILC)
- If a project EXISTS → we need to assess impact size

**Options:**
- (a) **No project exists** — this is a new initiative, needs its own project
- (b) **A project exists** — this is a change/feature for: {project name}___

**Your Decision:** _[awaiting input]_
```

#### If (a) — No project exists:
- Route = `new-project`
- Skip to Step 4 (produce Approved Idea Brief)

#### If (b) — Project exists:
- Proceed to Step 3 (Impact Assessment)

### Step 3: Lightweight Impact Assessment

**This is a quick determination, not a full feasibility study.** AI-PILC handles deep analysis if the change is big.

```markdown
### Q-{nn}: Impact Assessment

**Context:** I need to determine if this is a BIG change (needs project-level governance) or a SMALL change (can go straight to the build queue).

Answer these four questions about the approved idea:

1. **Scope impact:** Does this change the project's stated objectives or success criteria?
   → [Yes / No]

2. **Architecture impact:** Does this alter the architecture significantly? (new components, changed contracts, security model changes, data model changes)
   → [Yes / No]

3. **Resource impact:** Does this require budget or resource re-allocation beyond the current plan?
   → [Yes / No]

4. **Stakeholder impact:** Does this affect stakeholders beyond the immediate team? (new approvals needed, governance changes, external dependencies)
   → [Yes / No]
```

**Routing logic:**
- **ANY "Yes"** → BIG change → Route = `change-request` (→ AI-PILC change management)
- **ALL "No"** → SMALL change → Route = `feature-backlog` (→ AI-DLC backlog)

Present the determination:

```
Based on your answers:
- Scope impact: {Yes/No}
- Architecture impact: {Yes/No}
- Resource impact: {Yes/No}
- Stakeholder impact: {Yes/No}

**Routing determination:** {BIG CHANGE — AI-PILC change management / SMALL FEATURE — AI-DLC backlog}
**Rationale:** {because X impacts Y, this requires project-level governance / because no project-level criteria are affected, this can go directly to implementation}

Agree with this routing? [Yes / Override to {other route} / Explain more]
```

## Gate

**This is the final stage — no forward gate.** The workflow terminates after the brief is produced and the user confirms.

**Post-completion state:**
- `ilc-state.md` is read-only (no further modifications by AI-ILC)
- The brief becomes the **input artifact** for the successor package
- The state file becomes the **marker file** that successors detect

---

## Edge Cases

| Situation | Handling |
|-----------|----------|
| User disagrees with the routing determination | Allow override: "You have authority to route differently. Overriding to {route}. I'll adjust the brief format accordingly." Log the override in Decision Log. |
| Impact assessment is ambiguous (borderline) | Present honestly: "This is borderline — one 'Yes' answer, but a weak one. My recommendation is {X}. Override if you disagree." |
| User wants to route to AI-ADLC directly | Explain: "AI-ILC routes to AI-PILC or AI-DLC. If this needs architecture work, it goes through AI-PILC change management, which may then engage AI-ADLC. This preserves project governance." If user insists, log as an override. |
| Approval had conditions | Include conditions prominently in the brief under "Approval Conditions" — the successor must be aware of them |
| User wants to create multiple briefs (e.g., split into two features) | Supported: create two separate Feature Briefs, each scoped. Update register with both. Log in Decision Log. |
| The "project" for routing doesn't have AI-PILC/AI-DLC set up | Brief is still produced as a portable document. User can feed it manually to whatever process their project uses. Standalone value is maintained. |
