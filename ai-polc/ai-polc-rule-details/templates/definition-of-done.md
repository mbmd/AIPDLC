## Purpose

The Definition of Done defines what must be true about a completed increment BEFORE it ships. Increments that don't meet DoD cannot be released.

---

## Checklist

An increment is DONE when ALL of the following are true:

### Universal (applies to all increments)

- [ ] Code complete and peer-reviewed
- [ ] All acceptance criteria verified (pass/fail demonstrated)
- [ ] Tests written and passing (unit + integration as applicable)
- [ ] No P1/P2 bugs open in scope
- [ ] Documentation updated (if applicable)
- [ ] PO has reviewed and accepted

### Context-Adapted

{Include items triggered by context factors. Remove those that don't apply.}

- [ ] Compliance sign-off obtained; audit log entries verified (Compliance = Heavy)
- [ ] Contract tests passing; service health check verified (Microservices)
- [ ] Feature flag configured; rollback tested (Release Strategy = Continuous/Flags)
- [ ] Analytics instrumented; baseline measurement captured (Data-Driven = High)
- [ ] Security review completed; no new vulnerabilities (Security-sensitive)
- [ ] Accessibility tested against WCAG criteria (if UI-facing)

## Exception Process

| Step | Action |
|:---:|--------|
| 1 | Increment fails DoD check → flagged |
| 2 | PO assesses: Is the gap acceptable for this release? |
| 3 | YES → Record exception (POLC-D-NNN) with rationale + follow-up plan |
| 4 | NO → Increment is not shippable; fix required |
| 5 | Exception pattern tracking informs DoD revision |
