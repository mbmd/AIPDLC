## Purpose

The Definition of Ready defines what must be true about a backlog item BEFORE development begins. Items that don't meet DoR are not eligible for sprint planning.

---

## Checklist

A backlog item is READY when ALL of the following are true:

### Universal (applies to all items)

- [ ] Clear description of WHAT is expected (not implementation-specific HOW)
- [ ] Acceptance criteria defined and testable (Given/When/Then or equivalent)
- [ ] Goal/epic linkage documented (traceable to product goal)
- [ ] No unresolved dependencies blocking implementation
- [ ] Size estimated (story points or T-shirt size)

### Context-Adapted

{Include items triggered by context factors. Remove those that don't apply.}

- [ ] Bounded context identified (DDD architecture)
- [ ] Service ownership assigned and API contract defined (Microservices)
- [ ] Compliance requirements listed (Regulatory/Compliance = Heavy)
- [ ] Cross-team dependencies identified and resolved (Scale = Multi-team)
- [ ] Success metric defined and measurement instrumentation specified (Data-Driven = High)

## Exception Process

| Step | Action |
|:---:|--------|
| 1 | Item fails DoR check → flagged |
| 2 | PO assesses: Is the gap acceptable given context? |
| 3 | YES → Record exception (POLC-D-NNN) with rationale + time bound |
| 4 | NO → Item returns to backlog for refinement |
| 5 | Exception count tracked — pattern of exceptions = DoR needs updating |

## Review Cadence

| Trigger | Action |
|---------|--------|
| End of each release | Retrospective: Did DoR prevent rework? |
| >3 exceptions in one sprint | Evaluate: too strict or too vague? |
| Architecture change | Re-derive context-adapted items |
| Team feedback | Evaluate and modify if justified |
