## How It Works (3 Mechanisms)

### 1. Steering Files (Preventive — Before Code)
`.kiro/steering/` files are loaded into Kiro's context automatically. Rules encoded as steering become Kiro's default behavior — violations never get created.

### 2. Hooks (Detective — On Events)

### 3. Audit Agent (Corrective — On Demand)

## How Hooks Work

Hooks fire automatically based on IDE events. You'll see warnings ONLY when something is wrong. Silence = compliance.

**Debounce Strategy:**
- Security-critical hooks (secrets, tenant isolation, migrations) fire on EVERY file save
- Advisory hooks (naming, boundaries, coverage) fire ONCE at end of session

**If a hook warns you:**
1. Read the warning — it cites a specific rule ID
2. Find the rule in `.governance/rules/` for full context
3. Fix the issue OR request an exception (see below)

**If hooks are too noisy:**
See `.kiro/hooks/ENFORCEMENT-GUIDE.md` for which hooks to disable first (🟡 Advisory → remove first).

## How to Request a Rule Exception

If you need to formally bypass a rule (prototyping, phased delivery, legacy constraint):

1. Identify the rule ID (from the hook warning or audit report)
2. Document justification (why, scope, when you'll resolve it)
3. Get approval (different person for 🔴 Critical rules)
4. Log the exception in `compliance-log/exceptions/`
5. Set an expiry date (max 30 days Critical, 90 days High)

See `.governance/compliance-log/exception-workflow.md` for the full process.
