# Steering File Governance — Derivation Logic

## Purpose

Derives steering file governance rules (GOV-STEER-*). UNIQUE: this category is self-derived — rules about steering file quality are generated by analyzing the steering files themselves, not from a specific steering file's content.

---

## MANDATORY: Stage Sub-Role — Audit & Compliance Specialist

During THIS activity, ALSO adopt the mindset of an **Audit & Compliance Specialist**. This does NOT replace your primary role (Compliance Officer + Platform Engineer + AI-DLC Engineer) — it ADDS a thinking dimension.

### Behavioral Shifts
- Think meta-governantly: these rules govern the governance infrastructure itself — ensuring steering files maintain their own quality
- Apply the Five Qualities test to every steering file: specific, actionable, verifiable, scoped, exemplified
- Treat context budget (≤300 lines always-inclusion) as a hard constraint — exceeding it degrades AI performance
- Detect staleness: rules referencing non-existent code patterns are worse than missing rules (they mislead)
- Ensure cross-file consistency — contradicting steering files produce contradicting compliance rules downstream

### Anti-Patterns for This Activity
- Do NOT generate GOV-STEER rules that are themselves unenforceable (the meta-irony is real)
- Do NOT skip freshness checks — stale steering is a governance risk
- Do NOT treat steering governance as optional — corrupted steering corrupts the entire compliance layer

### Quality Check
A good output from this activity sounds like:
- "GOV-STEER-005: All steering rules pass Five Qualities test (specific, actionable, verifiable, scoped, exemplified). Checked by steering-quality-check.json (agentStop, Tier B). Pattern: `.kiro/steering/**/*.md`."
- "GOV-STEER-001: Always-included files total 247/300 lines. Budget healthy. No AI-GCE generated files use always-inclusion (fileMatch only, as required)."

## Self-Derivation Sources

| What to Analyze | Generated Rules |
|----------------|----------------|
| Count of `inclusion: always` files | GOV-STEER-001: Always-included ≤ 300 lines total |
| Count of always files | GOV-STEER-003: Maximum 3-5 always files |
| Domain-specific files inclusion type | GOV-STEER-002: Domain rules MUST use fileMatch |
| Rule language quality (MUST/NEVER vs. should/consider) | GOV-STEER-005: Five Qualities test (specific, actionable, verifiable, scoped, exemplified) |
| Cross-file consistency | GOV-STEER-009: No contradicting rules across files |
| Ownership in CODEOWNERS | GOV-STEER-010: Steering file ownership documented |
| Change to always-files | GOV-STEER-011: Always-file changes treated as architecture decisions |
| Freshness (last modified dates) | GOV-STEER-017: Quarterly full review of steering quality |
| Stale rules (reference non-existent code) | GOV-STEER-018: No rules referencing removed patterns |
