# Workspace Reading Guide

## Purpose

This document defines HOW AI-GCE reads, locates, and parses an AI-DWG development workspace to derive compliance rules and hooks. It covers detection strategy, file identification, content extraction patterns, and error handling when the workspace is incomplete or was not generated by AI-DWG.

---

## MANDATORY: Stage Sub-Role — Business Analyst

During THIS activity, ALSO adopt the mindset of a **Business Analyst**. This does NOT replace your primary role (Compliance Officer + Platform Engineer + AI-DLC v1 Engineer) — it ADDS a thinking dimension.

### Behavioral Shifts
- Focus on requirements elicitation: what does this workspace TELL us it needs?
- Classify each discovered file as a governance input, enrichment source, or irrelevant noise
- Think in terms of stakeholder needs: developers need hook patterns, auditors need traceability, maintainers need impact maps
- Prioritize completeness of discovery — missed files mean missed compliance rules later
- Document assumptions explicitly when workspace is incomplete rather than guessing

### Anti-Patterns for This Activity
- Do NOT invent requirements the workspace doesn't state (derive, never assume)
- Do NOT skip low-priority scan items — even infrastructure files confirm technology choices
- Do NOT treat absence of a file as permission to fabricate its content

### Quality Check
A good output from this activity sounds like:
- "Workspace scan found 14 steering files, 3 operational docs, confirmed NestJS/PostgreSQL stack from package.json + docker-compose.yml. Module-structure claims 6 modules; filesystem confirms 5 exist (flagging mismatch)."
- "Detection method: marker file found at .kiro/steering/workspace-rules.md. Proceeding with full Priority 1-5 scan."

## Step 1: Locate the Development Workspace

### Detection Strategy (Ordered)

```
1. User provides workspace path explicitly
   → Use that path directly

2. Assume current working directory
   → Check for .kiro/steering/workspace-rules.md

3. Scan common locations for the marker file:
   → ./
   → ../
   → Sibling folders of current directory

4. Marker not found
   → Ask user: "Where is the development workspace?"
   → User points to the project root

5. Partial workspace (steering exists but incomplete)
   → Proceed with what exists + built-in baseline
   → Warn about missing files that limit derivation depth
```

### Marker File: `.kiro/steering/workspace-rules.md`

This is the anchor. Once found, the workspace root is the PARENT of `.kiro/`. All other files are located relative to this root.

**Why this marker:** AI-DWG guarantees this file in every workspace it generates. If this file exists, AI-DWG has run (or someone manually created equivalent governance).

## Step 2: Scan Workspace Contents

### What to Scan (Ordered by Priority)

After locating the workspace root, scan for the following in order:

```
PRIORITY 1 — STEERING FILES (primary derivation source)
─────────────────────────────────────────────────────────
Scan: .kiro/steering/*.md
For each file found: record filename, read front-matter (inclusion type), note existence

PRIORITY 2 — OPERATIONAL DOCUMENTS (governance enrichment)
───────────────────────────────────────────────────────────
Scan for:
  • PROJECT_INSTRUCTIONS.md (project root)
  • DEFINITION_OF_DONE.md (project root)
  • TEAM_AGREEMENTS.md (project root)
  • CONTRIBUTING.md (project root)
  • CODEOWNERS (project root or .github/)

PRIORITY 3 — INFRASTRUCTURE FILES (technology confirmation)
─────────────────────────────────────────────────────────────
Scan for:
  • docker-compose.yml / docker-compose.yaml
  • package.json / pom.xml / *.csproj / Cargo.toml / go.mod
  • .gitignore
  • .editorconfig

PRIORITY 4 — FOLDER STRUCTURE (module path discovery)
──────────────────────────────────────────────────────
Scan actual source directories:
  • src/ or src/Modules/ or app/ or lib/
  • Record actual module folder names and paths
  • These drive hook file patterns (MUST use real paths, never guess)

PRIORITY 5 — EXISTING COMPLIANCE (detect prior AI-GCE run)
────────────────────────────────────────────────────────────
Check for:
  • .kiro/hooks/ (if populated → Mode 2 or Mode 4, not Mode 1)
  • .governance/ folder (if exists → compliance layer already installed)
  • .compliance-state.json (if exists → tier tracking active)
```

## Step 3: Read Steering Files — Extraction Rules

### Categorization by Derivation Type

When reading steering files, categorize each as ARCHITECTURAL or NON-ARCHITECTURAL. This determines which generator files to invoke and which rule prefixes to assign.

#### Architectural Steering (→ Architecture-derived rules)

| Steering File | Extract For | Rule Prefix |
|---------------|-------------|:-----------:|
| `workspace-rules.md` | Architecture identity, golden rules, DON'T rules | ARCH-* |
| `architecture-principles.md` | Numbered principles (P1-Pn) → each becomes a compliance check | ARCH-* |
| `tech-stack.md` | Technology names, versions, frameworks → drives ALL hook file patterns | — (cross-cutting) |
| `coding-standards.md` | Code patterns, formatting rules, DO/DON'T pairs | CODE-* |
| `api-standards.md` | REST conventions, URL patterns, error format, pagination, contract format | GOV-API-* |
| `security-rules.md` | Auth model, encryption rules, OWASP mitigations, sensitive data categories | SEC-* |
| `module-structure.md` | Module names, paths, layer rules, dependencies, ownership | MOD-* |
| `database-rules.md` | Schema patterns, migration rules, multi-tenant scoping, caching | DATA-* |
| `naming-conventions.md` | File/folder/class/method naming patterns | NC-* |
| `error-handling.md` | Error patterns (Result pattern, exception usage, API error format) | ERR-* |
| `observability-logging.md` | Log levels, message format, required logging points | LOG-* |
| `observability-sensitive.md` | What NEVER to log, masking rules, PII categories | SEC-* |
| `domain-context.md` | Ubiquitous language, bounded contexts, domain rules, entity names | DOM-* |

#### Non-Architectural Steering (→ Governance-derived rules)

| Steering File | Extract For | Rule Prefix |
|---------------|-------------|:-----------:|
| `role-isolation.md` | Roles defined, MUST NOT lists, segregation table, approval chains, scaling rules | GOV-ROLE-* |
| `session-governance.md` | Session rules (SG-01 to SG-12), session structure, autonomy mode | GOV-SESSION-* |
| `project-governance.md` | Sprint cadence, quality gates, escalation rules, phase gate criteria | PG-*, GOV-SPRINT-* |
| `git-workflow.md` | Branch strategy, commit convention, PR process, CI/CD stages, deployment gates, DR standards | GOV-DEVOPS-*, GOV-PR-* |
| `testing-strategy.md` | Coverage targets, test types required, CI/CD thresholds | GOV-CICD-* |
| `scope-and-risks.md` | Scope boundaries, risk categories → risk-adjusted enforcement severity | — (enrichment) |

#### Conditional Steering (→ Only if file exists)

| Steering File | Extract For | Rule Prefix |
|---------------|-------------|:-----------:|
| `multi-tenancy.md` | Isolation model, scoping strategy, cross-tenant prevention | TENANT-* |
| `api-versioning.md` | Version lifecycle, breaking change rules | API-VER-* |
| `resilience-standards.md` | Retry policies, circuit breaker config, timeout budgets | RES-* |
| `observability-tracing.md` | Span naming, required instrumentation points | TRACE-* |
| `performance-standards.md` | Response time budgets (p95/p99), measurement requirements | PERF-* |
| `workflow-engine.md` | State machine patterns, workflow lifecycle | WF-* |
| `frontend-standards.md` | Component patterns, accessibility requirements | FE-* |
| `event-sourcing.md` | Event store rules, projection patterns, CQRS boundaries | ES-* |
| `feature-flags.md` | Flag architecture, rollout strategies, lifecycle rules | FF-* |
| `brownfield-patterns.md` | Legacy module paths, characterization test requirements, migration guardrails | → Mode 3 |

## Step 4: Extract Content Patterns

### What to Extract from Each Steering File

For EVERY steering file, extract these element types:

| Element Type | What It Looks Like | Used For |
|--------------|-------------------|----------|
| **MUST/NEVER rules** | "MUST use Result<T>", "NEVER throw exceptions from domain" | Direct rule generation (one rule per MUST/NEVER) |
| **Table entries** | Rows in decision/convention tables | Rule parameters (thresholds, patterns) |
| **File patterns** | "*.controller.ts", "src/modules/*/presentation/" | Hook `patterns` field |
| **Named entities** | Module names, class patterns, folder conventions | Hook pattern specificity |
| **Thresholds/numbers** | "≥80% coverage", "p95 < 200ms", "max 3 contexts per team" | Rule severity + verification criteria |
| **DO/DON'T pairs** | Code examples showing right and wrong approach | Rule statement + anti-pattern field |
| **Role assignments** | "Approved By: Architect", "Session Owner ≠ Reviewer" | GOV-ROLE rule generation |
| **Phase references** | "Before Construction", "During Foundation" | Phase gate rules + hook phase-awareness |

### Technology Pattern Extraction (From `tech-stack.md`)

This is the MOST CRITICAL extraction because it determines ALL hook file patterns:

| tech-stack.md Says | AI-GCE Derives (File Patterns) |
|-------------------|---------------------------------|
| Language: TypeScript, Framework: NestJS | `*.controller.ts`, `*.service.ts`, `*.module.ts`, `*.entity.ts`, `*.dto.ts` |
| Language: Python, Framework: Django | `views.py`, `models.py`, `serializers.py`, `urls.py`, `admin.py` |
| Language: C#, Framework: ASP.NET Core | `*Controller.cs`, `*Service.cs`, `*Repository.cs`, `*.Entity.cs` |
| Language: Java, Framework: Spring Boot | `*Controller.java`, `*Service.java`, `*Repository.java`, `*Entity.java` |
| Language: Go | `*_handler.go`, `*_service.go`, `*_repository.go`, `*_model.go` |
| Database: PostgreSQL + EF Core | `Migrations/*.cs`, `*Configuration.cs` |
| Database: PostgreSQL + TypeORM | `src/migrations/*.ts`, `*.entity.ts` |
| Database: PostgreSQL + Django | `*/migrations/*.py`, `*/models.py` |

**Rule:** NEVER generate hooks with generic patterns like `**/*.ts`. ALWAYS derive specific patterns from tech-stack.md + module-structure.md.

### Module Path Extraction (From `module-structure.md` + Folder Scan)

| module-structure.md Says | AI-GCE Derives |
|-------------------------|----------------|
| Module: Incident, Path: src/modules/incident/ | Hooks watch `src/modules/incident/**` |
| Module: Finance, Path: src/Modules/Finance/ | Hooks watch `src/Modules/Finance/**` |
| Layer pattern: domain/application/infrastructure/presentation | Layer-specific hooks use `**/domain/**`, `**/presentation/**` |

**Rule:** Cross-reference module-structure.md content against ACTUAL folder scan. If module-structure.md says "src/modules/" but the actual folder is "src/Modules/" (case difference), use the ACTUAL path.

## Step 5: Read Operational Documents

### From `DEFINITION_OF_DONE.md`

Extract every DoD criterion → each becomes a verification item for the post-task-governance hook and phase gate rules.

| DoD Says | AI-GCE Generates |
|----------|-----------------|
| "All unit tests pass" | PG rule: tests must pass before task complete |
| "API contract updated" | PG rule: contract matches implementation |
| "Code reviewed by non-author" | GOV-ROLE rule: author ≠ approver |
| "No Critical security findings" | GOV-CICD rule: security scan = 0 Critical |

### From `TEAM_AGREEMENTS.md`

Extract PR process, commit conventions, review standards → feed into GOV-PR and GOV-ROLE rules.

| TEAM_AGREEMENTS Says | AI-GCE Generates |
|---------------------|-----------------|
| "Commit format: {type}({scope}): {description}" | GOV-PR rule: commit convention enforcement |
| "PR requires 1 CODEOWNER approval" | GOV-ROLE rule: CODEOWNER approval required |
| "No PRs > 500 lines without splitting" | GOV-PR rule: PR size limit |

### From `CODEOWNERS`

Extract module → owner mapping → feeds GOV-ROLE and GOV-TT rules.

| CODEOWNERS Entry | AI-GCE Generates |
|-----------------|-----------------|
| `src/Modules/Finance/** @finance-team` | GOV-TT rule: Finance module owned by finance-team |
| `.kiro/steering/architecture-*.md @architect` | GOV-STEER rule: architecture steering needs architect approval |
| `contracts/** @api-reviewer` | GOV-API rule: contract changes require api-reviewer |

### From `docker-compose.yml`

Extract infrastructure services → confirm database technology, detect environments.

| docker-compose.yml Shows | AI-GCE Confirms |
|-------------------------|----------------|
| `postgres:15` service | Database = PostgreSQL → migration patterns from database-rules.md |
| `redis:7` service | Caching layer exists → referenced in data-governance rules |
| Multiple profiles (dev/test/staging) | Environment strategy → GOV-DEVOPS environment rules |

## Step 6: Build the Derivation Context

After reading everything, compile the **derivation context** — the complete understanding used for rule generation:

```
DERIVATION CONTEXT:
├── Project Identity
│   ├── Name: {from PROJECT_INSTRUCTIONS.md or workspace-rules.md}
│   ├── Autonomy Mode: {from session-governance.md}
│   └── Team Size: {from role-isolation.md scaling section}
│
├── Technology Profile
│   ├── Language: {from tech-stack.md}
│   ├── Framework: {from tech-stack.md}
│   ├── Database: {from tech-stack.md + docker-compose.yml}
│   ├── File patterns: {derived from technology profile}
│   └── Layer pattern: {from module-structure.md}
│
├── Module Map
│   ├── {module-1}: path={actual path}, owner={from CODEOWNERS}
│   ├── {module-2}: path={actual path}, owner={from CODEOWNERS}
│   └── ...
│
├── Governance Profile
│   ├── Roles defined: {from role-isolation.md — count + names}
│   ├── Segregation rules: {from role-isolation.md — who ≠ who}
│   ├── Session rules: {from session-governance.md — SG-01 to SG-12}
│   ├── Sprint cadence: {from project-governance.md}
│   ├── Quality gates: {from project-governance.md}
│   ├── PR process: {from git-workflow.md + TEAM_AGREEMENTS}
│   ├── CI/CD thresholds: {from testing-strategy.md}
│   └── Phase gate criteria: {from project-governance.md + DoD}
│
├── Compliance Signals
│   ├── Multi-tenant: {yes/no — multi-tenancy.md exists?}
│   ├── Brownfield: {yes/no — brownfield-patterns.md exists?}
│   ├── Distributed: {yes/no — resilience-standards.md exists?}
│   ├── Event-sourced: {yes/no — event-sourcing.md exists?}
│   ├── Feature-flagged: {yes/no — feature-flags.md exists?}
│   ├── Performance-budgeted: {yes/no — performance-standards.md exists?}
│   ├── Frontend: {yes/no — frontend-standards.md exists?}
│   └── MCP configured: {yes/no — .kiro/settings/mcp.json has servers?}
│
├── Depth Assessment
│   ├── Steering file count: {n}
│   ├── Module count: {n}
│   ├── Conditional signals active: {n}
│   └── Determined depth: {Minimal | Standard | Comprehensive}
│
└── Built-in Baseline (ALWAYS applied regardless of above)
    ├── Spec before code
    ├── Never vibe code
    ├── Author ≠ approver
    ├── No direct push to main
    ├── Tests pass before merge
    ├── No hardcoded secrets
    ├── Migration has rollback
    ├── One task at a time
    ├── Compliance log append-only
    └── Critical exception needs different approver
```

This context drives ALL subsequent rule generation, hook creation, and agent configuration.

## Step 7: Detect Prior AI-GCE Run (Mode Selection)

| Finding | Mode |
|---------|------|
| No `.kiro/hooks/` or empty | Mode 1: Full Generation |
| `.kiro/hooks/` has content + no `.compliance-state.json` | Legacy state — treat as Mode 1 with merge |
| `.compliance-state.json` exists with `complianceTier: 1` | Mode 4 candidate (if user wants tier upgrade) |
| `.governance/brownfield-baseline.md` exists | Brownfield already baselined — Mode 2 or Mode 4 |
| brownfield-patterns.md exists but NO brownfield-baseline.md | Mode 3: Brownfield first-time adoption |

## Error Handling

| Situation | Response |
|-----------|----------|
| `.kiro/steering/workspace-rules.md` not found | Ask user for workspace location. If truly absent → can only apply built-in baseline (10 rules) |
| `tech-stack.md` missing or empty | WARN: cannot derive technology-specific file patterns. Use generic patterns as fallback. Mark hooks with `<!-- customize: tech-specific patterns needed -->` |
| `module-structure.md` missing | WARN: cannot derive module-specific paths. Scan actual folder structure as fallback |
| Steering file exists but has no extractable rules | Skip that category (no MUST/NEVER/table entries = nothing to derive from) |
| `CODEOWNERS` missing | WARN: cannot derive ownership-based rules (GOV-ROLE-015 through 019). Generate rules without ownership specifics |
| `TEAM_AGREEMENTS.md` missing | Use built-in baseline for PR process (author ≠ approver). Skip team-specific PR rules |
| Folder structure doesn't match module-structure.md | Use ACTUAL folders (trust filesystem over documentation). Flag mismatch in generation report |
| Conflicting rules between steering files | Flag conflict to user: "{file A} says X but {file B} says Y. Which takes precedence?" |
| Technology not in known pattern library | Generate generic hooks with `**/*.{extension}` pattern. Mark with `<!-- customize: {tech}-specific patterns needed -->` |
| Existing `.governance/rules/` with manual customizations | Detect `<!-- custom -->` tags. Preserve those rules during re-derivation. Never overwrite custom rules. |

## Key Rules

1. **NEVER invent governance decisions.** If a steering file doesn't state a rule, don't assume one. Apply built-in baseline for universal rules; skip project-specific derivation.
2. **Use ACTUAL paths, not assumed paths.** Always cross-reference module-structure.md against the real filesystem. The filesystem is the source of truth for paths.
3. **Technology specificity is non-negotiable.** A hook that watches `**/*.ts` is nearly useless. Derive `src/modules/*/presentation/**/*.controller.ts` from tech-stack + module-structure.
4. **Trace every derivation.** Every generated rule must state "Derived From: {steering file} → {specific section/table/row}."
5. **Respect conditional signals.** If multi-tenancy.md doesn't exist, tenant isolation rules are NOT generated. Period.
6. **Built-in baseline is the floor.** Even if ALL steering files are absent, the 10 baseline rules still get generated.
7. **Operational docs enrich, not replace.** TEAM_AGREEMENTS, CODEOWNERS, DoD enrich governance rules — they don't create new rule categories. The category exists from the steering file; the operational doc provides specifics.
8. **Phase-awareness starts at reading time.** Note which rules apply to which phase (Setup/Foundation/Construction/Integration/Go-Live) based on the steering file content. This feeds hook phase-checking.
