# Knowledge Map Guide

## Purpose

This document defines HOW to produce and maintain a **knowledge map** (traceability matrix) that connects every compliance rule to its source and its enforcement mechanism. The knowledge map answers: "Where did this rule come from, and what enforces it?"

---

## MANDATORY: Stage Sub-Role — Business Analyst

During THIS activity, ALSO adopt the mindset of a **Business Analyst**. This does NOT replace your primary role (Compliance Officer + Platform Engineer + AI-DLC v1 Engineer) — it ADDS a thinking dimension.

### Behavioral Shifts
- Think in traceability chains: every rule must connect source → enforcement → evidence
- Focus on stakeholder questions: developers ask "why does this fire?", auditors ask "is everything enforced?", maintainers ask "what breaks if I change X?"
- Treat gaps as findings, not omissions — an unenforced rule is a reportable gap
- Ensure the knowledge map serves as a living index, not just documentation
- Validate bidirectional traceability: steering → rule → hook AND hook → rule → steering

### Anti-Patterns for This Activity
- Do NOT leave rules without a source reference (orphan rules are traceability failures)
- Do NOT mark a rule as "enforced" without naming the specific mechanism
- Do NOT treat the knowledge map as static — it must update with every re-derivation

### Quality Check
A good output from this activity sounds like:
- "Knowledge map traces 142 rules: 128 enforced by hooks (90%), 8 by steering (preventive), 6 gaps flagged with severity and recommended action."
- "Reverse index: post-task-governance.json enforces 23 rules across 4 categories. Recommend splitting if this exceeds cognitive load threshold."

---

## What Is a Knowledge Map?

A knowledge map is a traceability matrix generated into the target workspace at `docs/knowledge-map.md`. It provides:

1. **For developers:** "Why does this hook fire? What's the source rule? Can I find the original decision?"
2. **For auditors:** "Is every governance commitment enforced? Are there gaps?"
3. **For maintainers:** "If I change a steering file, which rules and hooks are affected?"

## How to Generate the Knowledge Map

### Step 1: Inventory All Generated Rules

After generating `.governance/rules/*.md`, walk through every rule file and extract:
- Rule ID (e.g., `GOV-ROLE-004`)
- Rule title/description (one-line summary)
- `Derived From` field (steering file reference OR built-in baseline)

### Step 2: Map Rules to Enforcement Mechanisms

For each rule, determine WHICH mechanism enforces it:

| Enforcement Type | How to Detect | Example |
|-----------------|---------------|---------|
| **Hook** | Rule ID is cited in a hook's prompt text | GOV-ROLE-004 → segregation-check.json |
| **Steering** (preventive) | Rule is encoded in a steering file that Kiro reads pre-generation | SEC-01 → security-rules.md loaded before code generation |
| **Audit only** | Rule is only checked during periodic audit (no real-time hook) | GOV-SPRINT-001 (sprint plan exists) → periodic-audit.json |
| **Not yet enforced** | Rule exists but no hook or steering covers it | Mark as "⬜ Not yet enforced" |

### Step 3: Identify Gaps

A rule with no enforcement mechanism is a **traceability gap**. These must be:
- Flagged in the knowledge map: `⬜ Not yet enforced`
- Listed in the "Enforcement Gaps" section
- Prioritized: 🔴 Critical rules without enforcement = immediate action

### Step 4: Write the Knowledge Map

Generate `docs/knowledge-map.md` with:
1. Full traceability table (Rule → Source → Enforcement)
2. Summary statistics (total rules, enforced vs. gaps)
3. Enforcement gaps section
4. Source coverage section (which steering files have full coverage vs. partial)

## Knowledge Map Sections

### Section 1: Traceability Table

The core matrix. Every rule has exactly one row.

```markdown
| Rule ID | Description | Tier | Source | Enforcement |
|---------|-------------|:----:|--------|-------------|
| ARCH-01 | ... | 1 | steering/architecture-principles.md | Hook: post-task-governance |
```

**Column definitions:**
- **Rule ID:** The unique identifier (prefix-number format)
- **Description:** One-line summary of what the rule requires
- **Tier:** Which compliance tier activates this rule (1, 2, or 3)
- **Source:** Where the rule was derived from (steering file path OR "Built-in Baseline")
- **Enforcement:** Which mechanism checks it (Hook name, "Steering (preventive)", or "Audit only")

### Section 2: Summary Statistics

```markdown
## Summary

| Metric | Value |
|--------|-------|
| Total rules | {n} |
| Enforced by hook (real-time) | {n} ({pct}%) |
| Enforced by steering (preventive) | {n} ({pct}%) |
| Enforced by audit only (periodic) | {n} ({pct}%) |
| Not yet enforced (gaps) | {n} ({pct}%) |
| Rules from steering | {n} |
| Rules from built-in baseline | {n} |
```

### Section 3: Enforcement Gaps

```markdown
## Enforcement Gaps

Rules that exist but have no automated enforcement mechanism:

| Rule ID | Description | Severity | Recommended Action |
|---------|-------------|:--------:|-------------------|
| GOV-TT-006 | Enabling team has exit criteria | 🟡 | Add to periodic-audit scope |
| ... | ... | ... | ... |

**Priority:** Address all 🔴 Critical gaps before declaring AI-GCE generation complete.
```

### Section 4: Source Coverage

```markdown
## Source Coverage

How well each steering file is covered by enforcement:

| Steering File | Rules Derived | Rules Enforced | Coverage |
|---------------|:------------:|:--------------:|:--------:|
| security-rules.md | 12 | 12 | 100% ✅ |
| role-isolation.md | 10 | 8 | 80% 🟡 |
| session-governance.md | 8 | 8 | 100% ✅ |
| ... | ... | ... | ... |
```

### Section 5: Hook → Rule Mapping (Reverse Index)

```markdown
## Hook → Rules Checked

| Hook | Rules It Enforces |
|------|------------------|
| session-discipline.json | GOV-SESSION-002, GOV-SESSION-007, GOV-SESSION-008, GOV-SESSION-009 |
| security-gate-check.json | SEC-01, SEC-03, SEC-10 |
| segregation-check.json | GOV-ROLE-004, GOV-ROLE-005, GOV-ROLE-006 |
| post-task-governance.json | PG-*, GOV-SESSION-005, ARCH-01 |
| periodic-audit.json | ALL rules (full scan) |
| ... | ... |
```

This reverse index answers: "Which rules are checked by this hook?" — useful for understanding hook scope and identifying hooks that are overloaded.

## When to Generate/Update the Knowledge Map

| Event | Action |
|-------|--------|
| Mode 1 (Full Generation) | Generate complete knowledge map |
| Mode 2 (Re-Derivation) | Update affected rules' enforcement entries |
| Mode 4 (Tier Activation) | Update tier column for newly activated rules |
| After manual rule addition | Add row to traceability table (mark `<!-- custom -->`) |
| After hook addition/removal | Update enforcement column for affected rules |
| During periodic audit | Audit agent verifies knowledge map accuracy as part of Step 5 |

---

## Knowledge Map Maintenance Rules

1. **Every rule MUST have an entry** — no orphan rules allowed
2. **Source MUST be specific** — "steering/security-rules.md → Auth Model section", not just "security"
3. **Enforcement MUST list the specific mechanism** — hook filename, not just "hook"
4. **Gaps MUST be acknowledged** — "⬜ Not yet enforced" is honest; silent omission is not
5. **After removing a hook** — update all rules that referenced it; they become "Audit only" or gaps
6. **After changing a steering file** — verify the knowledge map still traces correctly
7. **Team-added custom rules** — tagged `<!-- custom -->` and traced to team decision (not steering)

## Integration with Other Components

| Component | Reads Knowledge Map For |
|-----------|------------------------|
| **Audit Agent** | Knows which rules to check (complete inventory) |
| **Compliance Dashboard** | Shows coverage statistics |
| **COMPLIANCE_README** | References knowledge map for "why does this rule exist?" |
| **ENFORCEMENT-GUIDE** | Hooks listed in knowledge map match available hooks |
| **Re-derivation** | Impact mapping: steering change → affected rules (uses source column) |

## Example: How a Rule Flows Through the Map

```
STEERING FILE: .kiro/steering/role-isolation.md
  └── Contains: "Segregation of Duties" table → "Session Owner ≠ Reviewer"
       │
       ▼
RULE GENERATED: .governance/rules/role-isolation.md → GOV-ROLE-004
  └── Derived From: role-isolation.md → Segregation + Built-in Baseline
       │
       ▼
HOOK ENFORCES: .kiro/hooks/segregation-check.json
  └── Prompt cites: GOV-ROLE-004, GOV-ROLE-005, GOV-ROLE-006
       │
       ▼
KNOWLEDGE MAP ENTRY:
  | GOV-ROLE-004 | Session Owner ≠ Reviewer | 2 | steering/role-isolation.md → Segregation + Built-in | Hook: segregation-check |
```

This is the full traceability chain: steering → rule → hook → knowledge map. An auditor can follow it in either direction.
