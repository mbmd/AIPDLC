# Hook Preamble — Package Territory Check

## Purpose

This file defines the **standard preamble** that MUST be prepended to every hook prompt generated by AI-GCE. It prevents hooks from firing against package infrastructure files — eliminating cross-package interference when AI-* family packages operate in the same workspace.

---

## The Standard Preamble (Copy Verbatim Into Every Hook Prompt)

```
## Package Territory Check (MANDATORY — evaluate FIRST)
Before performing any checks below, verify the triggering file path.

EXCLUDED ZONES (package infrastructure — NOT compliance subjects):
• .kiro/hooks/          — AI-GCE hook infrastructure
• .kiro/steering/       — AI-DWG/AI-GCE steering infrastructure
• .kiro/agents/         — AI-GCE agent infrastructure
• .governance/          — AI-GCE governance rules/agents/logs
• compliance-log/       — AI-GCE audit trail (append-only infrastructure)
• project-initiation/   — AI-PILC output (upstream documentation, not code)
• architecture/         — AI-ADLC output (upstream documentation, not code)
• management_framework/ — Shared governance spine
• docs/compliance-*     — AI-GCE generated dashboards
• templates/            — AI-DWG planning templates

If the triggering file's path starts with ANY of the above prefixes:
→ EXIT SILENTLY. Do NOT check, do NOT log, do NOT warn.
→ These files are package infrastructure and not subject to application compliance rules.

If the file is NOT in an excluded zone → proceed with checks below.
```

---

## When to Apply

| Hook Type | Apply Preamble? | Rationale |
|-----------|:--------------:|-----------|
| `fileEdited` (Tier A) | ✅ YES | File path is known — check before running logic |
| `fileCreated` | ✅ YES | File path is known — check before running logic |
| `agentStop` (Tier B) | ✅ YES | These hooks scan session-modified files — skip infra files in that scan |
| `promptSubmit` | ❌ NO | No file context — session-level check only |
| `preToolUse` | ❌ NO | Tool-type trigger, not file-based |
| `postTaskExecution` | ❌ NO | Task-level trigger, not file-based |
| `userTriggered` | ❌ NO | Manual trigger, scans by design (audit agent handles its own scope) |

**Exception — agentStop hooks:** These hooks scan ALL files modified during a session. The preamble instructs the AI to **skip infrastructure files** in that scan, not to skip the entire hook.

---

## Adaptation for Brownfield (Mode 3)

In brownfield workspaces, teams may have code in non-standard paths. The preamble's excluded zones list is generated from the **Package Territory Registry** (`PACKAGE_TERRITORIES.md`), not hardcoded. When AI-GCE generates hooks for a brownfield workspace:

1. Read `.governance/PACKAGE_TERRITORIES.md`
2. Include standard zones + any team-added custom exclusions
3. Populate the preamble with the combined list

---

## Integration with Hook Generation

When generating or re-deriving hooks (Step 4 of core-generator.md):

1. Load this file (`common/hook-preamble.md`)
2. For each hook that qualifies (see table above), prepend the preamble text to the beginning of `then.prompt`
3. The preamble goes BEFORE the phase check — it is the absolute first evaluation
4. For hooks that do NOT qualify (promptSubmit, preToolUse, postTaskExecution, userTriggered), skip the preamble entirely

---

## Source of Truth

The excluded zones in this preamble MUST match the zones declared in:
`templates/package-territories.md` (the template for `.governance/PACKAGE_TERRITORIES.md`)

If new packages are added to the AI-* family that produce output in new paths, update BOTH files:
1. `templates/package-territories.md` — the registry template
2. This file — the preamble text block

---

## Why This Design (Three-Layer Segregation)

This preamble is **Layer 2** of a three-layer segregation strategy:

| Layer | Mechanism | What It Catches |
|-------|-----------|-----------------|
| **Layer 1: Pattern Scoping** | Hook `patterns` arrays use scoped paths (e.g., `src/**/*.ts`) instead of bare `**/*.ts` | 90% of interference — structural prevention |
| **Layer 2: Prompt Preamble** (this file) | AI evaluates file path before running compliance logic | Edge cases that slip past Layer 1 |
| **Layer 3: Territory Registry** | Single source of truth for excluded zones, referenced by Layer 2 | Centralized, maintainable, extensible |

All three layers work together. Layer 1 prevents most false fires structurally. Layer 2 catches anything Layer 1 misses at runtime. Layer 3 keeps the exclusion list consistent and team-extensible.
