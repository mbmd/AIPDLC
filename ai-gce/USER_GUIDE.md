# AI-GCE — User Guide

**Package:** AI-GCE (AI-Driven Governance & Compliance Engine)
**Version:** 1.0.0
**Audience:** Tech Leads, DevOps Engineers, Engineering Managers, Compliance Officers, Team Leads

---

## What is AI-GCE?

AI-GCE is an adaptive governance engine that reads a development workspace and derives a tailored compliance enforcement layer — hooks, rules, agents, and logging infrastructure specific to that project's architecture, technology, team structure, and methodology. It enforces not just code patterns, but team topology, sprint governance, PR process, CI/CD gates, and session discipline.

**In one sentence:** AI-GCE turns your workspace into a self-enforcing compliance system — it derives the rules from what's already documented and enforces them automatically.

---

## When to Use AI-GCE

| Scenario | AI-GCE helps you... |
|----------|---------------------|
| Workspace generated by AI-DWG, need enforcement | Derive compliance hooks and rules from steering files |
| Team keeps drifting from architecture decisions | Automatic real-time enforcement on file save |
| Need compliance audit trail | JSONL logging of every hook execution |
| Existing project with no governance | Brownfield mode — baseline violations and enforce going forward |
| Compliance tier needs to increase (pre-release) | Progressive tier activation from 60% to 92%+ |

## How It Works (5 Minutes)

1. **Install** — Copy package files into your IDE's steering folder (see `setup/INSTALL.md`)
2. **Have a workspace** — Either generated by AI-DWG or manually set up with `.kiro/steering/` files
3. **Start generation** — Say: *"Using AI-GCE, generate the compliance engine for this workspace"*
4. **Answer 1-2 questions** — AI-GCE confirms team structure and initial tier
5. **Get your compliance layer** — Hooks, rules, agents, and logging — all derived in one pass

## Operating Modes

AI-GCE has four distinct modes — not a linear lifecycle:

| Mode | Trigger Phrase | What Happens |
|------|---------------|--------------|
| **1. Full Generation** | "Generate compliance engine" | One-shot derivation of all hooks, rules, agents from workspace |
| **2. Re-Derivation** | "Steering changed — re-derive" | Incremental update; preserves `<!-- custom -->` team additions |
| **3. Brownfield** | "Brownfield adoption" / "Baseline scan" | Scans existing violations; marks as tech debt; enforces new code only |
| **4. Tier Activation** | "Activate next compliance tier" | Increases enforcement level (Day 0 → Sprint 2+ → Pre-Release) |

---

## Three-Tier Progressive Compliance

AI-GCE never big-bangs compliance. Enforcement grows with the team:

| Tier | Coverage | When | What's Enforced |
|------|:--------:|------|-----------------|
| **Day 0** | 60-70% | Project kickoff | Essential rules only — security, module boundaries, session discipline |
| **Sprint 2+** | 80-90% | After team settles | Expanded — naming, API contracts, test conventions, PR process |
| **Pre-Release** | 92%+ | Before release | Full compliance — all rules active, zero tolerance on critical paths |

You control when to advance: *"Activate next compliance tier"*

## Enforcement Model

AI-GCE produces two types of enforcement:

### Hooks (Automatic, Real-Time)

Fire automatically on workspace events. No user action required.

| Hook Type | Fires On | Purpose |
|-----------|----------|---------|
| Security-critical | `fileEdited` (immediate) | Catches violations the moment they're written |
| Advisory/style | `agentStop` (batch) | Checks at end of AI session — less intrusive |

**Silent when passing.** Hooks produce zero output when rules are satisfied. You only see them when something is wrong.

### Process Agents (Manual, Milestone-Triggered)

User-invoked at process boundaries via shortcut triggers:

| Shortcut | Agent | When to Call |
|----------|-------|--------------|
| `SDC__` | Session Discipline Check | Before ending an AI session |
| `SGV__` | Sprint Governance Validator | At sprint boundary |
| `CRV__` | Code Review Validator | Before PR merge |
| `SQC__` | Security Quick Check | Before deploying |
| `CMG__` | Change Management Gate | Before major changes |
| `DOD__` | Definition of Done Audit | Before marking work complete |

---

## Derivation Model (Two-Source)

AI-GCE derives rules from two sources:

| Source | What It Provides |
|--------|-----------------|
| **Built-in baseline** | Universal governance rules (author ≠ approver, no direct-push, spec before code) |
| **Workspace steering files** | Project-specific rules derived from tech-stack, architecture, team topology |

Even with no steering files, the baseline provides meaningful governance. Steering files enrich and specialize.

## Brownfield Mode

Already have a codebase with violations? AI-GCE doesn't demand perfection from day one:

1. **Baseline Scan** — Identifies existing violations across the codebase
2. **Mark as Tech Debt** — Existing violations are logged but not enforced
3. **Enforce New Code** — From day 1, only new/changed code must comply
4. **Progressive Remediation** — Address tech debt at your pace

Say: *"Brownfield adoption — baseline scan"* to activate.

## Session Continuity

AI-GCE tracks state in `.compliance-state.json`:
- Current tier (Day 0 / Sprint 2+ / Pre-Release)
- Readiness criteria for next tier
- Baseline violations (brownfield)
- Custom rules registry
- Last derivation timestamp

Re-derivation preserves all team customizations marked `<!-- custom -->`.

## Tips for Best Results

1. **Run AI-DWG first (if possible)** — A well-structured workspace produces a richer compliance engine.
2. **Start at Day 0 tier** — Don't activate full compliance on a new team. Progressive adoption wins.
3. **Mark team additions `<!-- custom -->`** — This protects your rules during re-derivation.
4. **Use process agents at milestones** — They catch what hooks can't (process-level governance).
5. **Check compliance logs** — JSONL events are your audit trail. Review them periodically.
6. **Don't fight the silence** — If hooks produce no output, that means everything is passing.

## What AI-GCE Is NOT

- NOT a lifecycle — it's a governance engine (derive + enforce)
- NOT architecture design (that's AI-ADLC)
- NOT workspace generation (that's AI-DWG)
- NOT test governance (that's AI-TGE)
- NOT code implementation (that's AI-DLC v1)
- NOT a CI/CD tool — it generates rules that CI/CD can also check

AI-GCE is the **Compliance Guardian** — it answers *"Is the team following the architecture and process decisions we already made?"*
