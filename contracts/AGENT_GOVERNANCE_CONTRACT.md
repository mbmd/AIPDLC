## §1. Agent Types

Agents in the AI-* family fall into three categories:

| Type | Purpose | Invocation | Example |
|------|---------|------------|---------|
| **Infrastructure** | Project scaffolding, workspace setup — runs once or rarely | Explicit (first-time setup) | `project-init-agent` |
| **Process** | Governance checkpoints at workflow milestones — user invokes at process boundaries | Shortcut trigger (`XXX__`) | `session-discipline-agent`, `sprint-governance-agent` |
| **Audit** | Deep compliance inspection — produces scores, reports, remediation plans | Shortcut trigger or scheduled | `compliance-audit-agent` |

**Rule:** Every agent MUST be classified into exactly one type. Type determines cadence expectations and consequence severity.

## §2. Agent Registry

Every destination workspace that has AI-GCE (or AI-TGE) installed MUST contain:

```
.governance/AGENT_REGISTRY.md
```

This is the **single-source lookup** for all agents active in the workspace. Generated by AI-GCE during compliance engine generation (Step 5c), updated during re-derivation.

### Registry Format

```markdown

| GCE-AG-02 | project-init-agent | Infrastructure | (first-time only) | 1 | AI-GCE | Active |
| GCE-AG-03 | session-discipline-agent | Process | SDC__ | 1 | AI-GCE | Active |
| GCE-AG-04 | sprint-governance-agent | Process | SGV__ | 2 | AI-GCE | Active |
| GCE-AG-05 | code-review-agent | Audit | CRV__ | 2 | AI-GCE | Active |
| GCE-AG-06 | steering-quality-agent | Audit | SQC__ | 2 | AI-GCE | Active |
| GCE-AG-07 | dod-gate-agent | Process | DOD__ | 2 | AI-GCE | Active |
```

**Rules:**
- Use package-prefixed IDs: `{PKG}-AG-{NN}` (e.g., `GCE-AG-01`, `TGE-AG-01`). Each package numbers its own agents 01..N independently — no global coordination needed.

- Tier follows the same 1/2/3 progressive model as hooks (§6)
- The registry survives re-derivation — team-added rows (custom agents) are preserved via `<!-- custom -->` markers

## §3. Naming Convention

### Agent File Names

Pattern: `{domain}-{function}-agent.md`

| Component | Source | Examples |
|-----------|--------|----------|
| `{domain}` | The governance domain the agent covers | `compliance`, `session`, `sprint`, `code-review`, `steering`, `change`, `dod`, `test` |
| `{function}` | What the agent does (verb-noun or noun) | `audit`, `discipline`, `governance`, `quality`, `management`, `gate` |

**Examples:**
- `compliance-audit-agent.md`
- `session-discipline-agent.md`
- `sprint-governance-agent.md`
- `code-review-agent.md`
- `steering-quality-agent.md`
- `change-management-agent.md`
- `dod-gate-agent.md`

### Shortcut Trigger Names

Pattern: `{3-LETTER-UPPERCASE}__` (three capital letters + double underscore)

**Rules:**
- Letters must form a recognizable abbreviation of the agent's purpose
- Double underscore suffix is non-negotiable (consistent with the `IQA__` precedent)
- Must be unique across the entire workspace (no collisions)
- Registered in workspace-rules steering file for the AI to recognize

**Generated shortcuts (destination workspaces — AI-GCE output):**
- `SDC__` — Session Discipline Check
- `SGV__` — Sprint Governance Validation
- `CRV__` — Code Review Verification
- `SQC__` — Steering Quality Check
- `CMG__` — Change Management Gate
- `DOD__` — Definition of Done Gate
- `PRC__` — Pre-PR Readiness Check

## §4. Agent Anatomy (Required Sections)

Every agent `.md` file MUST contain the following structure. This is enforced during generation and validated during audit.

```markdown
---
name: {agent-id}
description: >
  {One to three line purpose statement}
generatedBy: {AI-GCE | AI-TGE | manual}
generatedVersion: {version}
source: {upstream-rule-path or "manual"}
generatedOn: {ISO-date}
ownership: generated
tools: ["{tool-categories}"]
trigger: {SHORTCUT__ or "manual" or "scheduled"}
tier: {1 | 2 | 3}
type: {infrastructure | process | audit}
---

# {Agent Title}

## Purpose
{What this agent checks/does — 2-3 sentences max}

## When to Invoke
{Explicit process milestone or cadence that triggers invocation}
{Include: "Call this when..." with concrete examples}

## Consequences of Skipping
{What degrades if this agent is never called}
{Include: immediate impact + accumulated debt over time}

## Recovery
{How to recover if the user forgot to invoke for N sessions/sprints}
{Include: specific recovery steps, not just "run the agent now"}

## Checks Performed
{Numbered list of what the agent evaluates}
1. ...
2. ...

## Output
{What the agent produces — log entries, reports, scores, decisions}
{Include: file paths where output lands}

## Related
{Cross-references to other agents, hooks, or steering files}
```

**Validation rule:** Any agent missing a required section fails the V7 structural check in `common/validation-rules.md`.

## §5. Producer Responsibility

Each package owns agents for **its own domain only**. Every package ships its own agents and registers them in the workspace when it runs. The governance contract guarantees no conflicts.

### Distributed Agent Model

**Principle:** When a user installs and runs any AI-* family package, that package:
1. Installs its own agents into `.kiro/agents/`
2. Appends its own shortcut block to `.kiro/steering/workspace-rules.md` (using `<!-- BEGIN/END -->` markers scoped by package name)
3. Registers its agents in `.governance/AGENT_REGISTRY.md` (creates the file if it doesn't exist; appends if it does)
4. Updates `.governance/AGENT-GUIDE.md` with its own section (creates if doesn't exist; appends its section if it does)

**No package depends on another package's agents being present.** Each package is self-sufficient for its domain.

### Conflict Prevention Rules

1. **Marker-based detection:** Before appending shortcuts, a package checks for its own `<!-- BEGIN {PACKAGE} AGENT SHORTCUTS -->` marker. If found → replace block. If not → append.
2. **Registry append-only:** Each package appends its rows to AGENT_REGISTRY.md using its own `{PKG}-AG-{NN}` prefix (e.g., `GCE-AG-01`). Each package numbers independently. Never re-number existing entries.
3. **No shortcut collisions:** Shortcuts are globally unique. Before assigning a new shortcut, check all registered shortcuts in workspace-rules. If collision → fail with error, don't overwrite.
4. **Domain scope in agent prompts:** Every agent's checks MUST be scoped to its domain. AI-PILC's agent checks PIP artifacts — never code. AI-GCE's agents check code/process — never PIP documents.

## §6. Tier Alignment

Agents follow the same three-tier progressive model as hooks:

| Tier | When Active | Agents Available (with triggers) |
|------|-------------|----------------------------------|
| **Tier 1** (Day 0) | From first generation | `project-init-agent` (first-time), `session-discipline-agent` (`SDC__`), `compliance-audit-agent` (manual), `test-governance-agent` (`TGV__`) |
| **Tier 2** (Sprint 2+) | After Tier 2 activation | + `sprint-governance-agent` (`SGV__`), `code-review-agent` (`CRV__`), `steering-quality-agent` (`SQC__`), `dod-gate-agent` (`DOD__`) |
| **Tier 3** (Pre-Release) | After Tier 3 activation | + `change-management-agent` (`CMG__`) |

**Rule:** An agent's `tier` field in front-matter determines when it appears in the AGENT-GUIDE and when its shortcut is registered. Tier 2/3 agents are INSTALLED but marked "(activates at Tier N)" until the workspace reaches that tier.

## §7. Lifecycle

### Creation
1. Package author creates the agent template in `templates/agents/`
2. Generator logic in `agents-from-steering.md` populates it with project-specific content
3. Generated agent is installed into `.kiro/agents/` in the destination workspace
4. Agent is registered in `.governance/AGENT_REGISTRY.md`
5. Shortcut is registered in workspace-rules steering file

### Versioning
- Agent version follows its parent package version (AI-GCE v1.0.0 → agents are v1.0.0)
- Re-derivation updates agent content but preserves `<!-- custom -->` sections (same as hooks)
- Version bump logged in AGENT_REGISTRY.md `lastUpdated` column

### Deprecation
- Mark agent `Status: Deprecated` in registry
- Remove shortcut from workspace-rules
- Keep the agent file for 1 major version (backward compatibility)
- Remove file in next major version

## §8. Cross-Package Coordination

### Self-Sufficiency Rule

Every package is independently installable. When a package runs, it MUST:
1. Create `.governance/AGENT_REGISTRY.md` if it doesn't exist (with header + its entries)
2. Append its entries if the file already exists (from a previously-run package)
3. Create `.governance/AGENT-GUIDE.md` if it doesn't exist (with header + its section)
4. Append its section if the file already exists
5. Never remove or modify another package's entries

### Marker-Based Ownership

Each package uses HTML comment markers to fence its content in shared files:

```markdown
<!-- BEGIN AI-PILC AGENT SHORTCUTS -->
... AI-PILC shortcut rules ...
<!-- END AI-PILC AGENT SHORTCUTS -->

<!-- BEGIN AI-GCE AGENT SHORTCUTS -->
... AI-GCE shortcut rules ...
<!-- END AI-GCE AGENT SHORTCUTS -->
```

**Re-derivation:** A package replaces ONLY its own block (between its markers). Other packages' blocks are untouched. This is how multiple packages coexist in one `workspace-rules.md`.

### Signaling Between Agents

Agents do NOT call other agents directly. If Agent A produces output that Agent B needs:
1. Agent A writes to a shared location (`.governance/compliance-log/`, `.compliance-state.json`, state files)
2. Agent B reads from that location when invoked
3. No runtime coupling — agents are independently invocable

### Conflict Resolution

If two agents could check the same thing (e.g., both `dod-gate-agent` and `compliance-audit-agent` check test coverage):
- The **specific** agent checks it as part of its focused workflow (DoD gate checks coverage as one of N criteria)
- The **general** agent reports on it as a metric (audit reports coverage score)
- Neither blocks the other — they produce complementary output

### Installation Order Independence

The packages can be installed in any order. The shared files (AGENT_REGISTRY, AGENT-GUIDE, workspace-rules) are designed for append-only growth:

| First Package Run | Creates | Format |
|---|---|---|
| Any (whichever runs first) | `.governance/AGENT_REGISTRY.md` | Header + first package's entries |
| Any subsequent | Appends to existing | Package's own `{PKG}-AG-{NN}` entries, no header rewrite |

**ID naming convention:** `{PKG}-AG-{NN}` — each package prefixes its own abbreviation and numbers its agents independently starting at 01. No global range allocation needed.

| Package | ID Prefix | Example |
|---------|-----------|---------|
| AI-ILC | `ILC-AG-` | `ILC-AG-01` |
| AI-PILC | `PILC-AG-` | `PILC-AG-01` |
| AI-ADLC | `ADLC-AG-` | `ADLC-AG-01` |
| AI-DWG | `DWG-AG-` | `DWG-AG-01` |
| AI-POLC | `POLC-AG-` | `POLC-AG-01` |
| AI-GCE | `GCE-AG-` | `GCE-AG-01` through `GCE-AG-09` |
| AI-TGE | `TGE-AG-` | `TGE-AG-01`, `TGE-AG-02` |
| AI-FLO | `FLO-AG-` | `FLO-AG-01` |
| AI-PPM | `PPM-AG-` | `PPM-AG-01` |
| AI-UXD | `UXD-AG-` | `UXD-AG-01` |
| Custom (team) | `CUST-AG-` | `CUST-AG-01+` |

**Why this works:** Each package owns its own counter — no range ceiling, no cross-file coordination, no collision risk. The prefix makes ownership immediately visible in the registry.

## §10. Consequences of Non-Invocation

This section defines the **governance debt model** — what accumulates when agents are not invoked at their designated process points.

### Debt Categories

| Category | Description | Detectability |
|----------|-------------|---------------|
| **Silent drift** | Standards erode without anyone noticing | Only visible in periodic audit scores |
| **Process skip** | A governance gate was bypassed | Logged as gap in compliance-log when audit runs |
| **Compliance gap** | Required evidence (sign-off, test, review) is missing | Blocks Tier advancement |

### Per-Agent Debt Accumulation

| Agent | Trigger | Debt If Skipped | Detection Window | Self-Healing? |
|-------|---------|-----------------|------------------|---------------|
| `session-discipline-agent` | `SDC__` | Vibe-coded work enters codebase without specs | Next audit detects missing spec files | ❌ Manual remediation (write specs retroactively) |
| `sprint-governance-agent` | `SGV__` | Sprint health unmeasured, retro actions lost | Next sprint boundary or audit | ⚠️ Partial — audit captures current state but can't reconstruct past sprint |
| `code-review-agent` | `CRV__` | Role isolation violations pass undetected | Audit agent checks git history | ⚠️ Partial — can flag past violations but can't undo merges |
| `steering-quality-agent` | `SQC__` | Stale/contradictory steering → hooks fire incorrectly | Audit or next `SQC__` invocation | ✅ Self-healing — checks current state regardless of when called |
| `change-management-agent` | `CMG__` | Changes reach production without sign-off | Post-deployment audit | ❌ Cannot undo deployment — requires exception + post-mortem |
| `dod-gate-agent` | `DOD__` | Incomplete tasks marked "done" → tech debt | Audit finds undone DoD items | ⚠️ Partial — can re-open but team may resist |

### Escalation Model

When a user repeatedly skips calling a process agent at its designated milestone, the compliance audit agent (safety net) detects the gap and escalates progressively:

| Missed Invocations | What Happens | User Experience |
|:------------------:|--------------|-----------------|
| **1–2 times** | Audit score dips slightly in the relevant category. No alarm raised. | You might not notice — the dashboard score decreases by a few percent. Normal for early adoption. |
| **3–5 times** | Audit explicitly flags "process adherence declining" in the compliance dashboard and in the next audit report. | You see a 🟡 warning on the dashboard: *"Sprint governance agent (`SGV__`) not invoked for 3 consecutive sprints."* |
| **6+ times** | Audit generates a formal **REMEDIATION entry** in `compliance-log/`. Team must acknowledge the gap and commit to resuming the cadence. | A remediation ticket appears: *"Process agent `SDC__` has not been invoked for 6+ sessions. Action required: team to acknowledge and resume cadence or formally waive via exception."* |

**Why this matters:** The escalation is not punitive — it's a visibility mechanism. The audit agent is the safety net that catches what process agents would have prevented. But audits are expensive (full scans) and reactive (detect after the fact, can't prevent). The earlier you call the process agent, the cheaper the governance.
