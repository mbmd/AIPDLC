## 1. Purpose

Define **one** protocol for safely upgrading AI-* packages in a workspace without losing user customizations to installed package files.

When a user installs an AI-* package (e.g., `ai-gce-rules/`, `ai-pilc-rule-details/`, `.kiro/hooks/*.json`), they may customize those files — tuning rules, adding team-specific hooks, editing templates. When a new version of that package ships, the update mechanism must:

1. **Detect** which installed files the user has modified
2. **Compare** the user's version against the incoming update
3. **Present** the differences clearly
4. **Let the user decide** (accept update / keep theirs / merge)
5. **Never silently overwrite** user customizations

> **This is a distribution/maintenance contract, not a project-change contract.** It governs how
> the AI-* packages *as tooling* update safely. Project artifact coherence (requirement changes
> flowing through the chain) is a separate domain — the planned **Lifecycle Change Propagation & Artifact Coherence** capability (idea `005-lifecycle-change-propagation`, not yet built).

---

## 2. Scope — What This Contract Covers

| In Scope | Out of Scope |
|----------|-------------|
| Package engine files (`ai-{pkg}-rules/`, `ai-{pkg}-rule-details/`) | Project artifacts generated by packages (those use `ownership` field + DWG reconciliation) |
| Hook templates (`.kiro/hooks/*.json`) installed by a package | User-authored hooks (never touched by package updates) |
| Package steering files installed into `.kiro/steering/` | Workspace-level steering the user wrote themselves |
| Agent templates (`.kiro/agents/*.md`) installed by a package | User-authored agents |
| INSTALL.md, README, LICENSE within the package | Files outside the installed package footprint |

**The boundary:** if a file has `generatedBy: AI-{PKG}` in its front-matter (or the JSON `generatedBy` field for hooks), it is package-managed and falls under this contract. Files without that marker are user-owned and never touched by updates.

## 3. The Two Versioning Dimensions

### 3.1 Dimension A — Artifact-Instance Safety (User Customizations)

The user has edited a package-managed file. Now an update wants to write a new version of that same file.

**Problem:** Silent overwrite destroys user work.
**Solution:** Three-way comparison (original install → user's current → incoming update).

### 3.2 Dimension B — Package-Version Awareness (Am I Current?)

The user wants to know if their installed packages are up-to-date with the latest published versions.

**Problem:** No mechanism to check "is AI-GCE v1.0.0 still current, or is v1.1.0 available?"
**Solution:** Installed-version tracking + online-version check protocol.

## 4. Installed State Tracking

Every installed AI-* package maintains a **package manifest** in the workspace:

```yaml
# .kiro/packages/ai-gce.manifest.yaml
package: AI-GCE
installedVersion: 1.0.0
installedOn: 2026-06-13T10:00:00Z
lastChecked: 2026-06-13T10:00:00Z
source: github.com/maheri/ai-gce
files:
  - path: ai-gce-rules/core-generator.md
    installedHash: abc123
    currentHash: abc123        # same = unmodified
    status: clean
  - path: ai-gce-rule-details/templates/hooks/naming-check.json
    installedHash: def456
    currentHash: xyz789        # different = user modified
    status: modified
  - path: .kiro/hooks/naming-check.json
    installedHash: ghi012
    currentHash: ghi012
    status: clean
```

### 4.1 Manifest Rules

| Rule | Description |
|------|-------------|
| **Created at install time** | The INSTALL process generates this manifest with hashes of every file written |
| **Location** | `.kiro/packages/{package-name}.manifest.yaml` |
| **One per package** | Each installed AI-* package has its own manifest |
| **Hash algorithm** | SHA-256 of file content (portable, fast) |
| **`installedHash`** | Hash at the moment the package was installed (the "original" baseline) |
| **`currentHash`** | Hash of the file right now (recalculated at check/update time) |
| **`status`** | `clean` (hashes match) / `modified` (user changed it) / `deleted` (user removed it) / `added` (user added file in package folder) |

## 5. Update Protocol — Dimension A (Safe Upgrade)

When a new version of a package is available and the user triggers an update:

### Step 1: Pre-flight Check

```
1. Load the package manifest (.kiro/packages/{pkg}.manifest.yaml)
2. Recalculate currentHash for every tracked file
3. Classify each file:
   - CLEAN: installedHash == currentHash (user hasn't touched it)
   - MODIFIED: installedHash != currentHash (user customized it)
   - DELETED: file no longer exists
   - ADDED: new file in package folder not in manifest
```

### Step 2: Categorized Update

| File Status | Update Behavior |
|-------------|----------------|
| **CLEAN** | Auto-update (safe — user hasn't customized, so no conflict) |
| **MODIFIED** | ⚠️ Present three-way diff: (a) original install, (b) user's current, (c) incoming update. User chooses: accept update / keep mine / merge manually. |
| **DELETED** | Ask: "You deleted `{file}`. The update includes a new version. Restore it or skip?" |
| **ADDED** | Skip — user-added files in package folders are never touched |
| **NEW in update** | Auto-add (new file that didn't exist before — no conflict possible) |

### Step 3: User Decision (for MODIFIED files)

Present per-file:

```markdown
### ⚠️ Conflict: ai-gce-rules/core-generator.md

You modified this file since installation. The update includes changes too.

**Your changes:** (summary of diff from installed → your current)
**Update changes:** (summary of diff from installed → new version)

Options:
(a) Accept update — your changes will be lost
(b) Keep yours — skip this file's update
(c) Merge — I'll show you both versions side by side for manual resolution
(d) View full diff — show me the three-way comparison
```

### Step 4: Apply & Record

```
1. Apply chosen resolution for each file
2. Update manifest:
   - installedHash = hash of the newly installed version
   - currentHash = hash of what's actually on disk (may differ if user kept theirs)
   - installedVersion = new package version
3. Log the update in the governance spine:
   - Decision Log entry: "{PKG}-D-NNN: Package updated from v{old} to v{new}. 
     {N} files auto-updated, {M} files user-resolved."
```

## 6. Version Check Protocol — Dimension B (Am I Current?)

### 6.1 Check-and-Notify (Not Auto-Update)

The user triggers a version check (manual command or periodic prompt):

```
1. For each installed package (one manifest per package):
   a. Read manifest → installedVersion
   b. Check configured source (repo/registry) for latest published version
   c. Compare: installedVersion vs. availableVersion
2. Report:
   
   Package Status:
   ✅ AI-PILC    v1.0.0  (current)
   ⚠️ AI-GCE     v1.0.0  → v1.1.0 available
   ✅ AI-DWG     v1.0.0  (current)
   ⚠️ AI-ADLC    v1.0.0  → v1.0.1 available (patch)
   
3. User chooses which (if any) to update
4. For each chosen → execute §5 Update Protocol
```

### 6.2 Version Source Configuration

Where the "latest version" comes from is **user-configured**, not hardcoded:

```yaml
# .kiro/packages/sources.yaml
defaultSource: github.com/maheri
packages:
  ai-gce:
    source: github.com/maheri/ai-gce
    branch: main
    checkInterval: weekly    # never | daily | weekly | manual
  ai-pilc:
    source: github.com/maheri/ai-pilc
    branch: main
    checkInterval: manual
```

**Rules:**
- No package auto-updates without user consent
- Check frequency is user-controlled (default: `manual`)
- Source can be any git repo, local path, or future registry
- If source is unreachable, report "unable to check" — never fail silently

## 7. Relationship to Existing Mechanisms

| Existing Mechanism | How This Contract Relates |
|--------------------|--------------------------|
| `ownership` field (NAMING_AND_OWNERSHIP §5.2) | Tells you who owns the file. This contract tells you what to DO during an upgrade for each ownership state. `ownership: generated` → auto-update safe. `ownership: hybrid` → present merge. `ownership: user` → never touch. |
| DWG `<!-- custom -->` markers | DWG's approach for project-artifact reconciliation (section-level preservation). This contract operates at the **file level** for package files — different scope, compatible approach. |
| `generatedBy` field | Used to identify which files belong to which package (the manifest tracks them). |
| GCE rederivation events | GCE re-derives compliance rules when workspace changes. That's project-level. This contract is about GCE *itself* updating (new rule templates in the package). |
| INSTALL.md per package | Currently handles fresh install only. This contract extends the INSTALL mechanism to cover upgrades. |

## 8. Package Obligations

Each package must support this contract:

| Obligation | Implementation |
|------------|---------------|
| **Ship a manifest template** | INSTALL process writes `.kiro/packages/{pkg}.manifest.yaml` with hashes of all installed files |
| **Version in state file** | The package's state file (e.g., `pilc-state.md`) includes `Producer Version: AI-PILC v{X.Y.Z}` (already done for most packages) |
| **Semantic versioning** | Follow semver: MAJOR (breaking), MINOR (additive), PATCH (fix). Determines update urgency. |
| **Changelog per release** | Ship a CHANGELOG.md in the package root documenting what changed per version |
| **Mark breaking changes** | MAJOR version bumps must list files that changed in incompatible ways — these get priority attention in the update protocol |

## 9. The `ownership` Integration

The `ownership` field from NAMING_AND_OWNERSHIP.md maps directly to update behavior:

| `ownership` value | Meaning | Update behavior |
|-------------------|---------|-----------------|
| `generated` | Fully tool-managed | Auto-update (file is not expected to be user-edited) |
| `hybrid` | Tool-seeded, team-edited | Three-way merge (§5 Step 3) — most common conflict case |
| `user` | Promoted to team ownership | **Never update** — tool treats as read-only. Report "skipped: user-owned." |

This means the manifest's `status: modified` flag and the `ownership` field work together:
- `ownership: generated` + `status: modified` → unusual (user edited something they shouldn't have). Warn more strongly: "This file is tool-managed. Your edits will be lost unless you change ownership to hybrid."
- `ownership: hybrid` + `status: modified` → expected. Standard merge workflow.
- `ownership: user` + any status → skip entirely.

## 10. Edge Cases

| Case | Handling |
|------|----------|
| **Manifest missing** (pre-contract install) | Treat all files as `status: unknown`. Offer to baseline: "I'll record current state as your baseline — no changes applied this time." |
| **File moved by user** | Manifest tracks by path. If path doesn't exist, mark `deleted`. User can re-map manually. |
| **Package adds new files in update** | Auto-add (no conflict possible — file didn't exist before) |
| **Package removes files in update** | Report: "The update no longer includes `{file}`. Keep your copy or remove it?" Never auto-delete. |
| **Multiple packages modify same file** | Should not happen (each package owns distinct files). If it does → flag as ownership conflict requiring user resolution. |
| **Offline / source unreachable** | "Cannot check for updates — source unreachable. Installed version: v{X.Y.Z}." Never fail the session. |

## 11. Boundaries (What This Contract Does NOT Do)

1. **Not project-artifact reconciliation.** DWG's delta reconciliation handles project-level steering file updates when architecture changes. This contract handles the package *tooling* updating.
2. **Not an auto-updater.** Check-and-notify only. User always decides when and whether to upgrade.
3. **Not a package registry.** Where packages are published and how they're distributed is out of scope. This contract assumes the package files are *already available* (from git clone, download, or future registry).
4. **Not dependency management.** If AI-GCE v1.1 requires AI-DWG v1.1, that's documented in the package's compatibility table — not enforced by this contract.
5. **Not rollback.** If an update goes wrong, the user re-installs the previous version. This contract doesn't maintain version history (git does that).
